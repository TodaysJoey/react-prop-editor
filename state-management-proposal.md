# Architecture Proposal: 상태 관리 라이브러리 도입을 통한 에디터 아키텍처 개선

## 1. 배경 및 현재 구조 (Context & Current Architecture)

현재 `react-prop-editor`는 최상위 컴포넌트인 [`App.tsx`](file:///Users/jihyun/Desktop/workspace/react-prop-editor/src/app/App.tsx)에서 React의 로컬 상태(`useState`)를 활용해 모든 에디터의 상태를 관리하고 있습니다.

```txt
┌─────────────────────────────────┐
│             App.tsx             │ (State: components, selectedComponentId)
└──────┬──────────────┬───────────┘
       │              │
       ▼              ▼
┌──────────────┐┌──────────────┐
│  Sidebar.tsx ││InspectorPane │
└──────────────┘└──────────────┘
```

이 구조는 규모가 작은 초기 단계에서는 직관적이고 자명하지만, 시각적 에디터(Visual Editor)의 특성상 기능이 고도화될수록 여러 아키텍처적 한계에 부딪히게 됩니다. 이를 해결하기 위해 중앙화된 상태 관리 라이브러리 도입을 제안합니다.

---

## 2. 현재 구조의 문제점 (Pain Points)

### 📌 2.1. 잦은 속성 변경에 따른 전체 리렌더링 및 성능 저하 (Performance)

- **현상:** 사용자가 `InspectorPane`에서 텍스트 필드를 입력하거나 색상, 크기 슬라이더를 조정할 때마다 매 글자/매 픽셀 단위로 최상위 `App`의 `setComponents`가 실행됩니다.
- **문제:** 단 하나의 프로퍼티 값 변경으로 인해 아무런 변화가 없는 `Sidebar`, `PreviewPane` 등 전체 컴포넌트 트리 구조가 매번 강제로 리렌더링됩니다.
- **결과:** 노드가 많아지거나 컴포넌트 내부가 무거워질 경우, 속성 편집기 타이핑이나 드래그 시 심각한 화면 끊김(Lag) 현상이 발생하게 됩니다.

### 📌 2.2. 과도한 Prop Drilling과 컴포넌트 간 높은 결합도 (Maintenance)

- **현상:** 에디터 하위 레이아웃 계층이 깊어지거나(예: Canvas 내의 섹션, 그룹 내부의 요소 등), 다단 툴바/레이아웃 트리 뷰가 추가될 경우 상태와 상태 변경 액션을 끝없이 하위 컴포넌트로 내려주어야 합니다.
- **문제:** `onSelectComponent`, `onChangeComponent`, `onDeleteComponent` 등의 콜백 프롭들이 거쳐 가는 중간 컴포넌트들을 오염시키고 유지보수를 어렵게 만듭니다.

### 📌 2.3. 비즈니스 로직과 UI 레이어의 혼재 (Separation of Concerns)

- **현상:** 새로운 노드 추가 시 고유 ID 생성 규칙, 노드 삭제 시 다른 노드로 선택 포커스를 옮기는 룰 등 에디터만의 핵심 도메인 로직이 React 컴포넌트인 `App.tsx` 내부에 함께 섞여 있습니다.
- **문제:** 코드의 관심사 분리가 되지 않아 코드를 읽기 어려워지고, 순수한 도메인 로직에 대한 단위 테스트 작성이 매우 곤란해집니다.

### 📌 2.4. 실행 취소 / 다시 실행 (Undo / Redo) 구현의 극심한 난이도

- **현상:** 시각적 에디터에서 절대 빠질 수 없는 핵심 기능이 바로 Undo/Redo(되돌리기/다시실행)입니다.
- **문제:** React의 `useState` 배열 구조 속에서 이전 모든 상태 스냅샷의 히스토리를 추적하고 얕은 복사/깊은 복사 버그 없이 이전 상태로 되돌리는 작업을 리액트 컴포넌트 내부에서 직접 구현하는 것은 극히 까다롭고 버그가 생기기 쉽습니다.

## 3. 제안하는 개선 방향 (Proposed Solution)

경량화된 전역 상태 저장소(Global State Store)를 도입하여 상태와 비즈니스 로직을 완벽히 분리하고, 필요한 컴포넌트만 정밀하게 상태를 구독하게 만듭니다.

```txt
          ┌───────────────────────┐
          │  Editor State Store   │ ◀─── [Read / Write via Hook]
          └───────────┬───────────┘
                      │
     ┌────────────────┼────────────────┐
     ▼                ▼                ▼
┌──────────────┐┌──────────────┐┌──────────────┐
│  Sidebar.tsx ││ PreviewPane  ││InspectorPane │
└──────────────┘└──────────────┘└──────────────┘
 (Only subscribes (Only subscribes (Only subscribes
   node list)     selected node)   selected props)
```

### ✨ 3.1. 구독형(Selector-based) 업데이트를 통한 렌더링 최적화

- 컴포넌트가 전체 상태를 다 들고 있는 대신, 자신에게 필요한 상태의 조각(예: `selectedNodeId` 혹은 특정 컴포넌트의 `props`)만 선택적으로 구독합니다.
- 이를 통해 인스펙터 창에서 값을 입력할 때, 해당 인스펙터의 개별 인풋과 변경이 발생한 프리뷰 노드 외의 무관한 컴포넌트들은 **리렌더링이 0회**로 억제됩니다.

### ✨ 3.2. 상태와 액션의 단일 저장소 관리 (Single Source of Truth)

- 상태 데이터와 이를 가공/수정하는 모든 함수(Actions)를 하나의 스토어 내부에서 캡슐화하여 관리합니다.
- 컴포넌트는 단지 `store.actions.updateComponentProps(...)`를 호출하기만 할 뿐, 구체적인 수정 방식이나 부가 로직은 스토어 내부에서 깔끔하게 처리됩니다.

### ✨ 3.3. 풍부한 에코시스템을 통한 Undo / Redo 기능의 가벼운 구현

- 검증된 상태 관리 라이브러리의 플러그인/미들웨어(예: Zustand의 `zundo` 등)를 결합하여, 단 몇 줄의 설정만으로도 완벽하게 격리되고 신뢰성 높은 Undo/Redo 스택을 장착할 수 있습니다.

## 4. 후보 라이브러리 및 선정 (Library Evaluation)

| 기준            | **Zustand (추천)**                                                                                                        | **Jotai / Recoil**                                       | **Redux Toolkit**                      |
| :-------------- | :------------------------------------------------------------------------------------------------------------------------ | :------------------------------------------------------- | :------------------------------------- |
| **개념**        | **Flux 아키텍처** 기반 단방향 캡슐화                                                                                      | **Atom (원자) 단위** 상향식 상태 관리                    | **정통 Flux / Boilerplate**가 큼       |
| **러닝 커브**   | 매우 낮음 (가장 직관적)                                                                                                   | 보통 (Atom 간 결합/파생 상태 이해 필요)                  | 높음 (코드가 장황해짐)                 |
| **렌더링 제어** | 수동 Selector 구독 방식으로 정밀 제어                                                                                     | Atom 구독 방식으로 자동 정밀 제어                        | Selector 구독 방식으로 정밀 제어       |
| **특장점**      | React 외부(Vanilla JS)에서도 상태 접근 가능, 가볍고 보일러플레이트가 거의 없음, **Undo/Redo 미들웨어(`zundo`) 완벽 지원** | React 동시성 모드 완벽 연동, 정교하게 분산된 상태에 유리 | 이미 대규모 앱에서 검증됨, 다소 무거움 |

### 💡 선정 제안: **Zustand (클라이언트 상태 관리의 최강자)**

시각적 컴포넌트 에디터는 대량의 단방향 데이터 흐름을 다루며, 무엇보다 **Undo/Redo(역사 관리)** 및 **쉬운 작성성**이 최우선 고려 대상입니다. Zustand는 보일러플레이트 코드가 거의 없고, `zundo` 라이브러리와의 유기적인 연동이 가능하여 에디터 설계에 최적의 선택지입니다.

## 5. 단계별 전환 로드맵 (Action Plan)

1. **라이브러리 추가:** `pnpm add zustand` (또는 `npm`/`yarn`) 실행
2. **스토어 뼈대 작성:** `src/libs/store.ts` 혹은 `src/features/editor/store.ts` 파일 정의
   ```typescript
   export const useEditorStore = create<EditorStore>((set) => ({
     nodes: [...],
     selectedNodeId: 'btn-1',
     actions: {
       selectNode: (id) => set({ selectedNodeId: id }),
       updateNodeProps: (id, props) => set(...)
     }
   }))
   ```
3. **최상위 컴포넌트 이관:** `App.tsx` 내의 로컬 `useState` 제거 및 스토어 연결
4. **하위 컴포넌트 구독 리팩토링:** `Sidebar`, `PreviewPane`, `InspectorPane`을 글로벌 스토어 훅 사용 방식으로 점진적 전환
5. **부가 기능 장착:** `zundo`를 통한 Undo/Redo 버튼 추가 및 단축키 바인딩
